# DuckDB Analysis — What Makes a Pokémon Type "Good"?

A sample data analysis built **entirely in [DuckDB](https://duckdb.org/) SQL**
on top of the same data that powers this project's PokéAPI Explorer
(`/api/v2/type`, `/api/v2/pokemon`, `/api/v2/pokemon-species`, …).

**The question:** if you could *be* a single Pokémon type, which one should you
pick? Two things decide it — how the **type-matchup chart** treats you, and how
much **raw bulk** Pokémon of that type actually carry. This notebook answers
both with SQL and then joins them for the payoff.

---

## Where the data comes from

The site reads the **live JSON API** at `https://pokeapi.co/api/v2`. DuckDB can
hit those exact endpoints directly once network egress to `pokeapi.co` is
enabled in the environment:

```sql
-- Live API: defensive profile of a single type (mirrors /api/v2/type/steel)
INSTALL httpfs; LOAD httpfs;
SELECT
  len(damage_relations.double_damage_from) AS weak_to,
  len(damage_relations.half_damage_from)   AS resists,
  len(damage_relations.no_damage_from)     AS immune_to
FROM read_json_auto('https://pokeapi.co/api/v2/type/steel');
-- → weak_to = 3, resists = 10, immune_to = 1   (matches the table below)
```

> **Note on this run:** this sandbox's egress policy blocks `pokeapi.co`, so the
> numbers below were generated by pointing DuckDB at PokéAPI's **published
> source CSVs** on GitHub
> (`raw.githubusercontent.com/PokeAPI/pokeapi/.../data/v2/csv/`). That is the
> *exact same dataset* the REST API serves — `type_efficacy.csv` is literally
> what `/api/v2/type` exposes as `damage_relations`. See
> [Reproducing this](#reproducing-this) at the end.

**Coverage of this run:** 1,025 default Pokémon · 18 standard types ·
324 type-matchup rows (the full 18×18 chart; `damage_factor ∈ {0, 50, 100, 200}`).

## Q1 — The matchup chart: who resists, who folds

`type_efficacy` stores a `damage_factor` for every attacker→defender pair:
`200` = super effective, `50` = resisted, `0` = immune, `100` = neutral. Reading
it **down the columns** gives each type's *defensive* profile.

```sql
WITH te AS (
  SELECT damage_type_id, target_type_id, damage_factor
  FROM type_efficacy
  WHERE damage_type_id BETWEEN 1 AND 18
    AND target_type_id BETWEEN 1 AND 18      -- 18 standard types only
)
SELECT t.identifier AS type,
  COUNT(*) FILTER (WHERE damage_factor = 200) AS weak_to,
  COUNT(*) FILTER (WHERE damage_factor = 50)  AS resists,
  COUNT(*) FILTER (WHERE damage_factor = 0)   AS immune_to,
  COUNT(*) FILTER (WHERE damage_factor = 50)
    + COUNT(*) FILTER (WHERE damage_factor = 0)
    - COUNT(*) FILTER (WHERE damage_factor = 200) AS net_defense
FROM te JOIN types t ON t.id = te.target_type_id
GROUP BY t.identifier
ORDER BY net_defense DESC, weak_to ASC;
```

|---|---|---|---|---|
| Steel | 3 | 10 | 1 | **8** |
| Poison | 2 | 5 | 0 | 3 |
| Fire | 3 | 6 | 0 | 3 |
| Electric | 1 | 3 | 0 | 2 |
| Fairy | 2 | 3 | 1 | 2 |
| Ghost | 2 | 2 | 2 | 2 |
| Water | 2 | 4 | 0 | 2 |
| Dragon | 3 | 4 | 0 | 1 |
| Flying | 3 | 3 | 1 | 1 |
| Normal | 1 | 0 | 1 | 0 |
| Bug | 3 | 3 | 0 | 0 |

resists just *one* type (itself), takes super-effective damage from four, and has
no immunities to soften the blow — the worst net spread of any type (**−3**).

## Q2 — Raw bulk: which types are actually built to take a hit?

Matchups are theory; base stats are the body. Here is the average **base-stat
total (BST)** and average **defensive bulk** (HP + Defense + Sp. Defense) per
*primary* type, across default forms only.

```sql
WITH bst AS (
  SELECT pokemon_id,
         SUM(base_stat)                                  AS total,
         SUM(base_stat) FILTER (WHERE stat_id IN (1,3,5)) AS bulk   -- HP+Def+SpDef
  FROM pokemon_stats WHERE stat_id BETWEEN 1 AND 6
  GROUP BY pokemon_id
),
prim AS (SELECT pokemon_id, type_id FROM pokemon_types WHERE slot = 1)
SELECT t.identifier AS type,
  COUNT(*)            AS pokemon,
  ROUND(AVG(total),1) AS avg_bst,
  ROUND(AVG(bulk),1)  AS avg_bulk_hp_def_spdef
FROM pokemon p
JOIN bst  b  ON b.pokemon_id = p.id
JOIN prim    ON prim.pokemon_id = p.id
JOIN types t ON t.id = prim.type_id
WHERE p.is_default = 1
GROUP BY t.identifier
ORDER BY avg_bst DESC;
```

|---|---|---|---|
| Dragon | 37 | 490.2 | 232.5 |
| Steel | 36 | 475.1 | **258.1** |
| Dark | 45 | 454.7 | 218.2 |
| Psychic | 60 | 446.7 | 220.9 |
| Fire | 66 | 446.2 | 206.2 |
| Fighting | 40 | 441.6 | 215.7 |
| Rock | 58 | 441.2 | 239.5 |
| Ice | 31 | 436.4 | 224.5 |
| Electric | 59 | 436.3 | 193.4 |
| Fairy | 29 | 436.1 | 228.1 |
| Flying | 9 | 436.1 | 204.8 |

**Read:** **Dragon** types average the highest overall power (490 BST), but

is the runt of the litter on every measure. Note also the population skew —
**Water** (134) and **Normal/Grass** are huge, common buckets, while **Flying**
is almost never a *primary* type (just 9 Pokémon lead with it).

## Q3 — The payoff: do good matchups and good bulk line up?

Joining Q1's matchup score to Q2's bulk reveals whether a type's reputation is
earned on *both* axes.

```sql
WITH te AS (
  SELECT target_type_id,
    COUNT(*) FILTER (WHERE damage_factor = 50)
      + COUNT(*) FILTER (WHERE damage_factor = 0)
      - COUNT(*) FILTER (WHERE damage_factor = 200) AS net_defense
  FROM type_efficacy
  WHERE damage_type_id BETWEEN 1 AND 18 AND target_type_id BETWEEN 1 AND 18
  GROUP BY target_type_id
),
bulk AS (
  SELECT pokemon_id, SUM(base_stat) FILTER (WHERE stat_id IN (1,3,5)) AS bulk
  FROM pokemon_stats WHERE stat_id BETWEEN 1 AND 6 GROUP BY pokemon_id
),
type_bulk AS (                 -- default forms only, primary type (matches Q2)
  SELECT pt.type_id, ROUND(AVG(b.bulk),1) AS avg_bulk
  FROM pokemon_types pt
  JOIN pokemon p ON p.id = pt.pokemon_id AND p.is_default = 1
  JOIN bulk b    ON b.pokemon_id = pt.pokemon_id
  WHERE pt.slot = 1
  GROUP BY pt.type_id
)
SELECT t.identifier AS type, te.net_defense, tb.avg_bulk
FROM te
JOIN types t      ON t.id = te.target_type_id
JOIN type_bulk tb ON tb.type_id = te.target_type_id
ORDER BY te.net_defense DESC, tb.avg_bulk DESC;
```

| type | net_defense | avg_bulk |
|---|---|---|
| Steel | **8** | **258.1** |
| Poison | 3 | 213.1 |
| Fire | 3 | 206.2 |
| Fairy | 2 | 228.1 |
| Ghost | 2 | 218.1 |
| Water | 2 | 209.4 |
| Electric | 2 | 193.4 |
| Dragon | 1 | 232.5 |
| Flying | 1 | 204.8 |
| Ground | 0 | 226.0 |
| Dark | 0 | 218.2 |
| Fighting | 0 | 215.7 |
| Normal | 0 | 200.5 |
| Bug | 0 | 189.4 |
| Rock | -1 | 239.5 |
| Psychic | -1 | 220.9 |
| Grass | -1 | 206.0 |
| Ice | **-3** | 224.5 |

### Meaningful impact

- **Steel wins on *both* axes simultaneously** — best matchup spread (+8) **and**
  the highest bulk (258). That double-confirmation is why a defensive core is so
  often built around a Steel type. The two independent data sources agree.
- **Ice is the trap type.** It has the worst matchups in the game (−3) and only
  *average* bulk (224) — nothing in its stats compensates for the chart. If you
  pick a type for survivability, this is the one to avoid.
- **Rock is a cautionary tale of one stat lying.** Rock carries the
  **2nd-highest bulk in the game** (239.5) thanks to sky-high Defense, yet its
  matchups are poor (−1, tied for 2nd-worst). Looking at the stat alone would
  have flattered it; the join with the matchup chart exposes the weakness.

## Q4 — Bonus: how much is a "Legendary" label actually worth?

```sql
WITH bst AS (
  SELECT pokemon_id, SUM(base_stat) AS total
  FROM pokemon_stats WHERE stat_id BETWEEN 1 AND 6 GROUP BY pokemon_id
)
SELECT
  CASE WHEN s.is_legendary = 1 THEN 'Legendary'
       WHEN s.is_mythical  = 1 THEN 'Mythical'
       ELSE 'Ordinary' END AS class,
  COUNT(*)            AS pokemon,
  ROUND(AVG(total),1) AS avg_bst,
  MIN(total)          AS min_bst,
  MAX(total)          AS max_bst
FROM pokemon p
JOIN pokemon_species s ON s.id = p.species_id
JOIN bst b             ON b.pokemon_id = p.id
WHERE p.is_default = 1
GROUP BY class
ORDER BY avg_bst DESC;
```

| class | pokemon | avg_bst | min_bst | max_bst |
|---|---|---|---|---|
| Legendary | 71 | 593.9 | 200 | 690 |
| Mythical | 23 | 586.1 | 300 | 720 |
| Ordinary | 931 | 411.1 | 175 | 670 |

**Read:** a Legendary averages **+182.8 BST** over an ordinary Pokémon — about
**1.44×** the power. But the label is *lore, not a stat floor*: the ranges
overlap hard. An ordinary Pokémon tops out at **670** (e.g. Slaking) — higher
than most Legendaries — while some "Legendaries" sit as low as **200** (weak
pre-evolutions). Useful caution if you ever filter a team by the `is_legendary`
flag expecting guaranteed strength.

## Caveats (so the numbers are read honestly)

- **Single-type matchups.** Real Pokémon often have two types, which *multiply*
  matchups (e.g. a 4× weakness). Q1/Q3 score each type in isolation — the
  standard way to compare types, and exactly what `/api/v2/type/{name}` returns.
- **Primary type only.** Q2/Q3 bucket each Pokémon by its `slot = 1` type to
  avoid double-counting; secondary types are ignored.
- **Default forms only** (`is_default = 1`): Mega Evolutions, Gigantamax, and
  regional forms are excluded so each species counts once.

## Reproducing this

```bash
# 1. DuckDB Python
pip install duckdb

# 2. Grab the PokéAPI source tables (the data behind the REST API)
BASE=https://raw.githubusercontent.com/PokeAPI/pokeapi/master/data/v2/csv
for f in types type_efficacy pokemon pokemon_types pokemon_stats stats pokemon_species; do
  curl -sO "$BASE/$f.csv"
done

# 3. Load them as views and run the queries above, e.g.
python3 -c "import duckdb; duckdb.sql(\"SELECT * FROM read_csv_auto('types.csv', header=true)\").show()"
```

When `pokeapi.co` egress is allowed, swap any `read_csv_auto('<table>.csv')` for
`read_json_auto('https://pokeapi.co/api/v2/<endpoint>')` to run the same logic
against the live API.
